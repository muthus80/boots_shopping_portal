## ADR-001 — Modular Monolith over Microservices

**Status:** accepted

**Context:** The application has 15 user stories across 6 functional domains with strong inter-domain dependencies (reviews require orders, checkout requires cart). The team is building a greenfield application targeting a 6-week delivery window with a single Docker Compose deployment target. Microservices would add distributed transaction complexity (saga pattern for checkout), inter-service networking overhead, and operational burden that is not justified at initial scale.

**Decision:** Adopt a modular monolith architecture where each domain (auth, products, cart, checkout, reviews, account) is a Python package with its own routes, services, and repositories — but all deployed as a single FastAPI process. Module boundaries are enforced by directory structure and import discipline, not network calls.

**Alternatives considered:**
- Microservices — rejected due to distributed transaction complexity for cart-to-order conversion and reviews purchase-verification, plus excessive operational overhead for a 6-week MVP
- Traditional monolith (single module) — rejected because it inhibits future extraction and makes domain boundaries ambiguous for multiple developers
**Consequences:**
- All inter-domain calls are in-process function calls with zero latency penalty
- Single deployment unit simplifies Docker Compose and CI/CD pipeline
- SQLAlchemy transactions span multiple domains without saga complexity
- Future migration to microservices is possible by promoting modules to services with well-defined REST interfaces
_Related: US-001, US-003, US-008, US-009, US-011_
## ADR-002 — JWT with Refresh Token Rotation for Session Management

**Status:** accepted

**Context:** US-013 requires session persistence across page refreshes and new tabs, with graceful redirect to login on expiry. The user preference specifies no dedicated auth service (auth: none in preferences means no third-party auth provider). We need a stateless auth mechanism that supports the browser's multi-tab scenario while allowing token revocation for logout. Storing hashed refresh tokens in PostgreSQL provides revocability without a separate Redis cache.

**Decision:** Issue short-lived (30-minute) JWT access tokens and long-lived (7-day) refresh tokens. Store a SHA-256 hash of the refresh token in the refresh_tokens table with a revoked flag. The React frontend stores the access token in memory and the refresh token in an httpOnly cookie. On access token expiry, silently call POST /api/v1/auth/refresh before retrying the original request.

**Alternatives considered:**
- Session cookies with server-side session store — rejected because it requires Redis or DB session store and is less suitable for a SPA architecture
- Long-lived JWTs without refresh — rejected because they cannot be revoked on logout, violating security requirements
- OAuth2 provider (e.g., Auth0) — rejected because the user specified auth: none (no third-party auth provider)
**Consequences:**
- Access tokens are stateless and verifiable without DB lookup on every request
- Logout and token revocation are supported via the revoked flag in refresh_tokens table
- httpOnly cookie for refresh token prevents XSS access to the long-lived token
- Token refresh adds one extra API call when access token expires, but this is transparent to the user
_Related: US-002, US-013, NFR-security-password-hashing_
## ADR-003 — Stripe PaymentIntents API with Stripe Elements for PCI-DSS Compliance

**Status:** accepted

**Context:** The critical NFR requires PCI-DSS compliance ensuring no cardholder data is stored on application servers. The checkout must be completable in 5 steps or fewer (US-011). We need a payment integration that handles card data entirely within the Stripe iframe while giving the backend control over order amounts.

**Decision:** Use Stripe PaymentIntents API on the backend to create server-authorised payment intents with the correct order total, and Stripe Elements (CardElement React component) on the frontend for card capture. The frontend confirms the payment directly with Stripe using the client_secret; the backend then verifies payment_intent status before creating the order record.

**Alternatives considered:**
- Stripe Checkout (redirect) — rejected because it takes the user off-site, breaking the 5-step checkout UX requirement
- Braintree — rejected because Stripe is the only provider explicitly referenced in the NFRs and user preference
- Manual card collection + server-side Stripe charge — rejected because it places card data on application servers, violating PCI-DSS
**Consequences:**
- Application servers never see raw card data — achieves PCI-DSS SAQ A compliance
- PaymentIntent server-side creation ensures the amount is set by the backend, preventing client-side tampering
- Two API calls required for checkout (create PaymentIntent + confirm), but both fit within the 5-step UX constraint
- Stripe webhook handling should be added post-MVP for reliable payment status updates
_Related: US-011, NFR-security-pci_
## ADR-004 — PostgreSQL Full-Text Search for Product Discovery

**Status:** accepted

**Context:** US-004 requires keyword search across product name, brand, and description. US-005 requires faceted filtering by size, color, and price range. The chosen database is PostgreSQL. Introducing a separate search engine (Elasticsearch, Typesense) would add operational complexity and a new Docker container without clear benefit at initial product scale.

**Decision:** Use PostgreSQL tsvector and tsquery for full-text search on products. Maintain a search_vector TSVECTOR column updated via database trigger combining name, brand, and description fields. Use a GIN index on search_vector for sub-100ms query performance. Filtering uses standard SQL JOINs to product_variants with indexed columns.

**Alternatives considered:**
- Elasticsearch — rejected due to operational overhead, additional container, and index synchronization complexity for an initial-scale product catalog
- Typesense — rejected for same reasons as Elasticsearch; overkill for a single-store catalog
- ILIKE/LIKE substring search — rejected because it does not use indexes and would fail LCP performance NFR under load
**Consequences:**
- Full-text search is available within PostgreSQL with no additional infrastructure
- GIN index on tsvector provides fast search at catalog scale
- If catalog grows to millions of SKUs or requires typo tolerance, migration to a dedicated search engine is straightforward
- tsvector trigger must be updated when product fields change
_Related: US-004, US-005, NFR-performance-lcp_
## ADR-005 — React 18 SPA with React Query for Server State Management

**Status:** accepted

**Context:** The frontend needs to satisfy US-014 (loading, empty, error states), US-015 (keyboard accessibility), and perform well enough to meet the LCP 2.5s NFR. The user specified React as the frontend framework. A SPA architecture serves the dynamic filtering (US-005) and cart state updates (US-010) without full page reloads.

**Decision:** Use React 18 SPA with React Query (TanStack Query) for server state, Axios as the HTTP client, and React Hook Form for all form handling. Tailwind CSS for styling. All forms implement ARIA labels; interactive components use semantic HTML (button, input, nav) for keyboard accessibility.

**Alternatives considered:**
- Next.js — considered for SSR performance benefits but rejected because the user explicitly specified react (not next.js) and SSR adds deployment complexity beyond Docker Compose
- Redux Toolkit — rejected in favour of React Query which handles server state, caching, and loading/error states more ergonomically with less boilerplate
**Consequences:**
- React Query provides built-in loading/error/empty state management satisfying US-014 with minimal custom code
- SPA routing via React Router v6 enables fast client-side navigation
- No SSR means first-paint depends on CDN-cached static JS bundle; CDN delivery is required to meet LCP 2.5s NFR
- Stripe Elements integrates cleanly as a React component within the checkout flow
_Related: US-014, US-015, NFR-performance-lcp, NFR-accessibility_
