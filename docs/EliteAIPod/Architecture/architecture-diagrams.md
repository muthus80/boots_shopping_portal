## Sequence Diagrams

### User Registration Flow

End-to-end sequence for new user registration: form submission, email uniqueness check, bcrypt hashing, user creation, and confirmation email dispatch.
```mermaid
sequenceDiagram
    participant B as Browser
    participant F as FastAPI
    participant DB as PostgreSQL
    participant E as Email Service

    B->>F: POST /api/v1/auth/register {email, password}
    F->>F: Validate email format + password complexity
    F->>DB: SELECT user WHERE email = ?
    DB-->>F: No row found
    F->>F: bcrypt.hash(password, rounds=12)
    F->>DB: INSERT INTO users (email, hashed_password)
    DB-->>F: User row created
    F->>E: Send confirmation email to user
    E-->>F: 202 Accepted
    F-->>B: 201 {user_id, email, message}
```

### Guest Checkout with Stripe Payment

Full checkout sequence: shipping info collection, server-side PaymentIntent creation, client-side card capture via Stripe Elements, payment verification, and order record creation with confirmation email.
```mermaid
sequenceDiagram
    participant B as Browser
    participant F as FastAPI
    participant DB as PostgreSQL
    participant S as Stripe API
    participant E as Email Service

    B->>F: POST /api/v1/checkout/payment-intent
    F->>DB: Fetch cart items + calculate total
    DB-->>F: Cart data + total_amount
    F->>S: Create PaymentIntent {amount, currency}
    S-->>F: {client_secret, payment_intent_id}
    F-->>B: 200 {client_secret}

    B->>S: Stripe Elements confirmCardPayment(client_secret)
    S-->>B: Payment confirmed

    B->>F: POST /api/v1/checkout/confirm {payment_intent_id}
    F->>S: Retrieve PaymentIntent (verify status=succeeded)
    S-->>F: PaymentIntent {status: succeeded}
    F->>DB: BEGIN TRANSACTION
    F->>DB: INSERT order + order_items
    F->>DB: DELETE cart + cart_items
    F->>DB: COMMIT
    DB-->>F: Order created
    F->>E: Send order confirmation email
    E-->>F: 202 Accepted
    F-->>B: 201 {order_id, order_number}
```

### Product Search and Filter Flow

Sequence for keyword search and faceted filtering: full-text search using PostgreSQL tsvector, and conjunctive filter query via variant joins.
```mermaid
sequenceDiagram
    participant B as Browser
    participant F as FastAPI
    participant DB as PostgreSQL

    B->>F: GET /api/v1/products/search?q=hiking
    F->>DB: SELECT products WHERE search_vector @@ plainto_tsquery('hiking')
    DB-->>F: Matching product rows
    F-->>B: 200 {products: [...], total: N, no_results: false}

    B->>F: GET /api/v1/products?category=work-boots&size=10&color=brown&max_price=150
    F->>DB: SELECT p FROM products p JOIN categories c ON c.slug='work-boots' JOIN product_variants v ON v.product_id=p.id WHERE v.size='10' AND v.color='brown' AND p.price <= 150
    DB-->>F: Filtered, paginated rows
    F-->>B: 200 {products: [...], total: N, page: 1}
```

### JWT Session Refresh Flow

Shows how the React frontend silently refreshes an expired access token using the refresh token, maintaining session continuity without redirecting to login.
```mermaid
sequenceDiagram
    participant B as Browser
    participant F as FastAPI
    participant DB as PostgreSQL

    B->>F: GET /api/v1/account/orders (expired access token)
    F-->>B: 401 Unauthorized

    B->>F: POST /api/v1/auth/refresh {refresh_token}
    F->>DB: SELECT refresh_token WHERE token_hash=? AND revoked=false AND expires_at > NOW()
    DB-->>F: Valid refresh token row
    F->>F: Sign new access JWT (exp: 30min)
    F-->>B: 200 {access_token, expires_in: 1800}

    B->>F: GET /api/v1/account/orders (new access token)
    F->>DB: SELECT orders WHERE user_id=?
    DB-->>F: Order rows
    F-->>B: 200 {orders: [...]}
```
