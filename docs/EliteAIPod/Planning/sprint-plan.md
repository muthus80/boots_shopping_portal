## Risks

- **[CRITICAL]** Stripe payment integration complexity: incorrect PaymentIntent lifecycle handling, webhook failures, or Stripe Elements misconfiguration could block the checkout flow entirely and delay the Sprint 3 milestone.
  - _Mitigation:_ Allocate dedicated spike time at Sprint 3 start to validate Stripe test-mode integration end-to-end before building the full UI. Use Stripe's official React library (stripe-js) and test all card decline and network error scenarios with Stripe test cards.
- **[HIGH]** Sprint 3 scope is now larger after absorbing US-008 (reviews) from Sprint 2. T-019 (Reviews API) depends on both T-018 (reviews schema) and T-025 (orders schema), both of which must complete before T-019 can start. If T-025 slips within Sprint 3, T-019 and T-020 are blocked.
  - _Mitigation:_ Schedule T-025 (orders schema) and T-018 (reviews schema) as the first db_architect tasks in Sprint 3. Monitor their completion at Sprint 3 day 2 standup. T-025 and T-018 have no intra-sprint dependencies and can run in parallel.
- **[HIGH]** WCAG 2.1 AA keyboard accessibility remediation (US-015) spans the entire application and may surface more defects than estimated, particularly in third-party components like Stripe Elements.
  - _Mitigation:_ Integrate accessibility checks (axe-core, keyboard testing) from Sprint 1 onwards rather than treating it as a Sprint 3 activity only. Use T-033 and T-034 as a final audit pass, not the first pass.
- **[MEDIUM]** LCP 2.5s performance requirement on product listing and detail pages may be violated by unoptimised high-resolution boot images.
  - _Mitigation:_ Implement lazy loading, next-gen image formats (WebP/AVIF), and CDN delivery for product images from Sprint 2. Include LCP measurement in the Sprint 2 definition of done for T-011 and T-017.
- **[MEDIUM]** Guest cart and authenticated cart merge: when a guest user logs in after adding items to cart, cart state reconciliation logic is complex and may cause data loss or duplicate items.
  - _Mitigation:_ Define cart merge strategy (merge guest items into user cart on login) explicitly in T-022 acceptance criteria. Cover merge edge cases in backend service tests.
- **[MEDIUM]** Email delivery for registration confirmation (US-001) and order confirmation (US-011) depends on a third-party transactional email provider (e.g., SendGrid, Postmark) which is not yet selected.
  - _Mitigation:_ Select and configure the transactional email provider before Sprint 1 development begins. Use environment-variable-driven configuration so the provider can be swapped without code changes.
- **[LOW]** T-028 (global state components) now depends on T-011 and T-013 completing in Sprint 2. If those tasks slip, T-028 cannot be completed and Sprint 2 pages will lack consistent error/loading/empty states.
  - _Mitigation:_ T-011 and T-013 are mid-sprint deliverables with no complex dependencies beyond their API tasks. Prioritise T-009, T-010, T-012 early in Sprint 2 to unblock T-011 and T-013 quickly, enabling T-028 to complete before sprint end.
